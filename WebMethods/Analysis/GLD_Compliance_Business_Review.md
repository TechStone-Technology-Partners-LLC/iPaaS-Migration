## What Does This Integration Do?

When a customer applies for a GLD product (loan, account, line of credit, etc.), the bank is legally required to run a **compliance check** through an external system called the **CIU (Compliance Identity Unit)**. This integration is responsible for everything that happens in the *database* around that compliance check:

- Saves the customer's information before the check is sent
- Archives the raw request for audit purposes
- Stores the reference number returned by CIU
- Records whether the check passed or failed
- Logs any errors that occur
- Retrieves full customer context when a response comes back
- Cleans up old records on a maintenance schedule

> **In short:** This integration is the data tracking layer for the compliance check workflow. It does not make the compliance decision — it records everything before, during, and after the decision is made by the external CIU system.

## Integration Flow — Step by Step

```
┌─────────────────────────────────────────────────────────────────────┐
│                  GLD COMPLIANCE CHECK — DATA FLOW                   │
└─────────────────────────────────────────────────────────────────────┘

  [1] RECEIVE COMPLIANCE REQUEST
      ↓  Customer details arrive (name, address, SSN/TIN, DOB, product info)

  [2] SAVE REQUEST TO DATABASE
      ↓  All customer + application fields written to Oracle database
         → Database returns a unique internal tracking ID (Request ID)

  [3] ARCHIVE THE RAW REQUEST
      ↓  Full XML of the request saved for audit trail
         → No output; this is write-only for compliance record-keeping

  [4] SEND TO EXTERNAL COMPLIANCE SYSTEM (CIU)
      ↓  Request forwarded to the external CIU compliance service
         → CIU returns a CIU Reference Number

  [5] SAVE CIU REFERENCE NUMBER
      ↓  The CIU reference number is written back to the database
         (links our internal record to the external CIU record)

  [6] WAIT FOR COMPLIANCE RESULT
      ↓  CIU sends back pass/fail result asynchronously

  [7] LOOK UP FULL CUSTOMER RECORD
      ↓  Full customer + request details retrieved using CIU reference

  [8] RECORD THE RESULT
      ├── PASS →  Log the passing result with Check Type and Reference Number
      └── FAIL →  Log the error details (type, code, description, reference)

  [9] END PROCESS
      └── [Maintenance path: scheduled purge of old compliance records]
```

## The 7 Database Operations — Plain English

| # | Operation Name | What It Does | Data In | Data Out |
|---|---|---|---|---|
| 1 | Log Check Request | Saves a new compliance check request to the Oracle database | Customer name, address, DOB, SSN/TIN, product details (25 fields) | Internal Request ID |
| 2 | Log Check Request XML | Saves the raw XML of the request for the audit trail | Application ID, full XML string, 3 identifiers | Nothing (write-only) |
| 3 | Log Check Reply | Records the compliance check result | CIU Reference Number, Check Type, Pass/Fail result | Nothing (write-only) |
| 4 | Log Check Reply Error | Records error details if the compliance check fails | Error type, error code, error description, CIU Reference | Nothing (write-only) |
| 5 | Select Customer And Request | Retrieves full customer + request record for a given CIU reference | CIU Reference Number | All 28 customer + request fields |
| 6 | Update CIU Reference Number | Saves the reference number returned by CIU onto the original request | Internal Request ID, CIU Reference Number | Nothing (write-only) |
| 7 | Purge Data | Deletes old/expired compliance records on a maintenance schedule | None | Nothing (housekeeping) |

## Key Business Decisions to Review and Approve

> Please review each item and confirm or flag for discussion.

### ✅ Confirm: Integration Scope
- This integration handles **database operations only** — it does not replace or replicate the CIU compliance system itself
- The external CIU endpoint URL is **not defined** in this package — it must be provided separately before the Boomi process can be completed
- The flow logic (triggering, routing, retry logic) comes from the companion package `GLDComplianceCheck`, which is a **separate migration effort**

### ⚠️ Review Required: PII Data Handling
- **SSN/TIN and Date of Birth** are stored in the Oracle database as plain text via these stored procedures
- In Boomi, process logging will be **disabled** to prevent SSN/TIN from appearing in integration logs
- Confirm: Does the target database (post-migration) maintain the same encryption/masking controls as the source Oracle database?

### ⚠️ Review Required: External CIU System
- Step 4 (Send to CIU) requires an external compliance service endpoint
- That endpoint is **not defined** in this package — it was handled by a separate webMethods service in the original system
- **Action needed:** Provide the CIU service URL, authentication method, and request/response format

### ✅ Confirm: Database Destination
- All 7 operations target the same Oracle database: `CSC06DSHORA1S:1522` (schema: `GLD_SCHEMA`)
- Confirm: Is this the same database in the target environment, or is there a new database host?
- The database password is **not stored in the integration** — it will be configured separately via secure environment settings

### ⚠️ Review Required: Data Type Mapping
Two fields require transformation that should be verified:

| Field | Issue | Resolution |
|---|---|---|
| Date of Birth (DOB) | webMethods stores as text string `yyyy-MM-dd`; Oracle expects a DATE type | Boomi will convert the format automatically using a built-in date function |
| Compliance Result | webMethods passes this as a true/false Boolean; the Oracle procedure expects the text `'TRUE'` or `'FALSE'` | Boomi will convert this before calling the database |

### ✅ Confirm: Record Purge Behavior
- The `Purge Data` operation calls a stored procedure with **no input parameters**
- The date threshold for what gets purged is **built into the Oracle stored procedure** on the database side — it is not configurable from this integration
- Confirm: Is the current purge threshold correct, or does it need to be adjusted in the database?
